= Known Problems with ESP8266 Development

== Boot Problems with CH340C USB-to-Serial Converter

=== Problem Description
- ESP8266 uploads firmware successfully but fails to boot normally after reset
- Shows garbled serial output instead of running the uploaded program  
- GPIO0 voltage drops from 3.3V to ~1.4V when serial monitor connects

=== Root Cause
The serial monitor automatically asserts DTR/RTS control lines when connecting, which interferes with the ESP8266 boot mode control circuit. This keeps GPIO0 in an intermediate voltage state, preventing normal boot mode.

=== Solution
Add these settings to your PlatformIO environment configuration:

[source,ini]
----
[env:your_esp8266_env]
platform = espressif8266
board = esp07
framework = arduino

# Other settings...

upload_resetmethod = ck          # Better for CH340-based boards
monitor_dtr = 0                  # Prevent monitor from asserting DTR
monitor_rts = 0                  # Prevent monitor from asserting RTS
monitor_filters = direct
----

=== Hardware Considerations
- Ensure GPIO0 has a proper pull-up resistor (4.7kΩ recommended for CH340C)
- GPIO2 should have 10kΩ pull-up to 3.3V
- GPIO15 should have 10kΩ pull-down to GND

=== Testing Procedure
1. Upload firmware
2. Wait 2-3 seconds before starting serial monitor
3. GPIO0 should maintain 3.3V and device should show proper serial output

=== Notes
This issue commonly affects custom PCBs with CH340C converters where the auto-reset circuitry interferes with normal boot sequence. The problem manifests as the ESP8266 getting stuck in an intermediate boot state due to improper GPIO0 voltage levels.
